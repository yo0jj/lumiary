## 1. 제품 개요 & 전략

### 1-1. 핵심 컨셉

하루 끝, 일기를 쓰려면 ① 앱을 열고 ② 뭘 쓸지 정하고 ③ 꾸준히 해야 한다. **전화 포맷은 이 세 장벽을 동시에 뒤집는다.**

| 기존 일기 앱 장벽 | 여보세요의 해결 |
| --- | --- |
| ① 앱을 *내가* 열어야 함 | AI가 먼저 전화를 건다 (Push, not Pull) |
| ② 빈 화면에 *뭘 쓸지* 정해야 함 | 4블록 질문 프레임이 대화를 이끈다 |
| ③ *꾸준히* 해야 함 | 실천 루프로 기록이 삶에 개입한다 |

### 1-2. 전략 위치

- 팀 one-liner: *재미로 시작한 기록 → 습관 → 일상 속 실천*
- 본 제품 = **진입(편리·재미) + 지속(습관)을 음성 통화로, 전환(실천)을 통화 끝 미니 액션 루프로** 커버

### 1-3. 핵심 플로우

```
[온보딩 — 최초 1회]
가입 → 호칭·목소리·시간 설정 → 0일차 미니 통화

[매일 반복]
설정 시간 → 푸시 알림 → 풀스크린 수신 오버레이
→ 받기(길이 선택) / 나중에
→ 인앱 음성 대화 (4블록 프레임, 1~7분)
→ 통화 종료 → 데일리 카드 생성 → 푸시
→ 카드 확인 → 별점/칩 평가

[다음 날 통화 첫마디]
"어제 {실천 과제} 해봤어요?" → 회고 → 새 실천 세팅
```

### 1-4. 실천 루프 (핵심 차별점)

```
[통화 ④블록] 내일의 아주 작은 실험 1개 함께 정함
     ↓
[다음 날 통화] "어제 그거 해봤어요?" 첫마디 확인
     ↓
성공 → 칭찬 + 기록     실패 → 더 작게 쪼개줌(난이도 조정)
```

> 단순 "말 잘 들어주는 AI"가 아니라 **기록 → 실천 → 회고 → 기록**의 닫힌 루프로 삶에 개입.

### 1-5. MVP의 목적 (가장 중요)

> **이 MVP는 "완성된 제품"이 아니라 "가설 검증 도구"다.** 가장 적은 빌드로 가장 리스크 큰 가설(습관·깊이)을 검증한다. 정교한 분석·디바이스·캐릭터는 전부 후순위.

## 2. 가설 & 성공 지표

### 2-1. 핵심 가설

> 가설은 *측정 가능한 형태*로. "검증 방법"과 "성공 기준" 없으면 가설이 아님. **H1·H2가 킬러 가설**, H3~H5는 보조.

| # | 분류 | 가설 | 검증 방법 | 성공 기준(가설값) |
| --- | --- | --- | --- | --- |
| H1 | **습관/지속** | AI가 먼저 전화하면 결심 없이도 꾸준히 기록한다 | 2주 베타 통화 응답률·리텐션 | 응답률 ≥ 60%, D7 리텐션 ≥ 40% |
| H2 | **깊이/거부감** | 음성으로 하면 글보다 거부감 없이 솔직하게 말한다 | 사용자 발화량 + 사후 칩 + 인터뷰 | "솔직했다" ≥ 70%, 평균 발화량 충분 |
| H3 | 재미/신박함 | "일기가 전화 건다"는 경험이 또 받고 싶게 만든다 | 만족도 + "또 받고 싶다" 칩 | 만족도 ≥ 4.0/5, "또 받고싶다" ≥ 70% |
| H4 | 편리함 | 말하기가 타이핑보다 부담이 적다 | 완주율 + "더 편했다" 칩 | 완주율 ≥ 80%, "더 편했다" ≥ 70% |
| H5 | 실천(라이트) | 통화 끝 실천 제안이 다음 행동으로 이어진다 | 수락률 + 다음날 응답률 | 수락 ≥ 50%, "해봤다" ≥ 30% |

### 2-2. 검증 환경

- **폐쇄형 베타**: 8~15명, 2주간 매일 통화
- **방법**: 다이어리 스터디(매일 사용) + 시작/종료 1:1 인터뷰
- 양적 지표(§2-3) + 질적 인사이트 병행
- 1차 판단 질문 = "숫자가 X% 넘었나"보다 **"이 사람들이 자발적으로 또 받고 싶어 하는가"**

### 2-3. 지표 (Metrics)

**North Star (MVP 한정):** 주간 1인당 완료 통화 수 (Weekly Completed Calls per User) — "꾸준한 기록"의 가장 직접적 대변.

| 퍼널 | 지표 |
| --- | --- |
| **진입** | 온보딩 완료율 · 첫 통화 수신율 · 첫 통화 완주율 |
| **습관 ⭐** | 통화 응답률(핵심) · D1/D3/D7 리텐션 · 연속 기록일 분포 |
| **깊이 ⭐** | 평균 통화 시간/발화량 · "솔직했다" 응답률 · 자기명료도 변화(정성) |
| **만족** | 통화 후 만족도 · 요약 카드 정확도("내 얘기 맞아?") |
| **실천** | 실천 제안 수락률 · 다음날 응답률 |
| **가드레일** | 통화 중 이탈률 · 마이크 권한 거부율 · 부정 피드백(어색/소름) 비율 · **안전 트리거 발생 건수** |

> ⚠️ 목표 수치는 전부 **가설값**. 베타 시작 전 팀이 baseline으로 고정 → 종료 후 비교.

## 3. 범위 (MoSCoW)

| 우선순위 | 기능 |
| --- | --- |
| **Must (MVP)** | 온보딩(호칭·목소리·시간·권한) · 푸시 트리거 · 인앱 콜 수신 오버레이 · 음성 통화(실시간 자막 + 텍스트 토글) · 4블록 대화 · 데일리 카드 · 만족도(별점+칩) · 홈 · 기록+Transcript · 마이/설정 · **위기 신호 안전 분기** |
| **Should** | 직전 통화 참조 변주 · 내일의 실천 제안 + 데일리 카드 노출 · 0일차 미니 통화 · 연속 기록일(streak) · 미수신 1회 재알림 |
| **Could** | 다음날 실천 확인 루프(첫마디) · 목소리 톤 3종 · 통화 길이 선택 · 회전 렌즈 다양화 |
| **Won't (Out-of-scope)** | 실제 PSTN 전화(Twilio) · 카카오톡 폴백 · 주간 리캡/무드 카드 공유 · 친구 공유·소셜 · AI 분신·캐릭터 · 정량 감정 그래프/패턴 분석 · 결제 · 멀티 시간대 · 다국어 |

> **실천 루프 스코프 노트:** "제안 + 카드 노출"은 Should, "다음날 첫마디 확인"은 Could. 리소스 빠듯하면 확인 루프를 베타 후반에 붙이는 옵션 → §14-A1.

## 4. 타겟 유저 (베타)

- 자기 자신을 관찰·기록하고 싶은 욕구가 있는 **20~30대**
- 일기/기록을 시도했지만 *지속에 실패한* 경험
- 말로 푸는 걸 글보다 편하게 느끼는 성향(모집 시 선별)
- **제외**: 정신과적 위기 상태 사용자 — *이 제품은 상담 대체가 아님*

## 5. 디자인 원칙 & 언어

### 5-1. 디자인 원칙

1. **조용한 존재감** — 하루 끝에 옆에 있는 느낌. 화려함보다 포근함.
2. **수동적 참여** — 사용자가 뭘 해야 한다는 압박 없음. 받고 말하면 끝.
3. **빛의 은유** — 자기 인식이 쌓일수록 밝아지는 시각 피드백.
4. **죄책감 없는 흔적** — 스트릭 깨져도 "쉬어가도 괜찮아요". 경쟁 아닌 자기 돌봄.

### 5-2. 컬러 팔레트

| 역할 | 색상 | 용도 |
| --- | --- | --- |
| Primary | `#2D4A3E` 딥 그린 | 주요 버튼, 강조 |
| Primary Light | `#4A7C6F` | 보조 강조 |
| Background | `#F8F5F0` 따뜻한 오프화이트 | 전체 배경 |
| Surface | `#FFFFFF` | 카드, 모달 |
| Text Primary | `#1A1A1A` | 본문 |
| Text Secondary | `#6B6B6B` | 부제목, 메타 |
| Text Tertiary | `#ABABAB` | 플레이스홀더, 비활성 |
| Accent | `#F2C94C` 소프트 옐로우 | 별점, 키워드 하이라이트 |

### 5-5. 스트릭 워딩 원칙

- ❌ "3일 연속 성공!" → ✅ "나를 만난 날 3일째"
- ❌ "오늘도 못 했네요" → ✅ "쉬어가도 괜찮아요, 내일 또 전화할게요"

## 6. 정보 구조 (IA)

```
여보세요
├── 온보딩 (최초 1회)
│   ├── Step 1: 호칭 입력
│   ├── Step 2: AI 목소리 톤 선택
│   ├── Step 3: 전화 시간 선택
│   └── Step 4: 0일차 미니 통화 → 완료
│
├── 수신 오버레이 (전화 올 때 — 앱 상단 덮음)
│   ├── 풀스크린 수신 UI
│   └── 통화 중 UI
│
├── 홈 (Tab)            오늘 상태 배너 · 최근 카드 1개 · → 카드 상세
├── 기록 (Tab)          대화 기록 리스트(최신순) · → 카드 상세
└── 마이 (Tab)
    ├── 프로필 + 통계
    └── 설정 (전화 시간 / 목소리 톤 / 호칭 변경)
```

> ❌ 제거: 전화번호 입력·변경 (인앱 푸시라 불필요).

## 7. 화면별 상세 스펙

### [S-01] 스플래시 · 1.5초 자동 이동

```
[전체 배경: Primary #2D4A3E]
  (수직 중앙)
  [여보세요 워드마크 — 흰색]
  [Caption] "나에게 걸려오는 대화형 일기"
```

**이동:** 신규 → S-02 / 기존 → 홈

### [S-02] 온보딩 Step 1 — 호칭 입력

```
[Progress 1/4]
  [Heading 1] 어떻게 불러드릴까요?
  [Body/Secondary] AI가 전화에서 이 이름을 불러요.
  [입력창] placeholder "이름 또는 별명" · maxLength 20
  하단 고정 [Primary] 다음  ← 1글자 이상 시 활성
```

---

### [S-03] 온보딩 Step 2 — AI 목소리 톤 선택

**목적:** 작은 개인화로 초기 애착 형성. *(톤 3종은 음성 스택 결정에 종속 — §11/§14)*

```
[Progress 2/4] [← 뒤로]
  [Heading 1] AI 목소리를 골라주세요
  [Body/Secondary] 나중에 언제든 바꿀 수 있어요.

  [선택 카드 × 3]
  ┌──────────────────────────┐
  │ 🌿 차분한   편안하고 조용한 │
  ├──────────────────────────┤
  │ ☀️ 명랑한   밝고 에너지 있는 │
  ├──────────────────────────┤
  │ 🪨 담백한   군더더기 없는   │
  └──────────────────────────┘
  [미리 듣기] 각 카드 하단 — 3초 샘플
  하단 고정 [Primary] 다음  ← 선택 시 활성
```

**카드 상태:** 미선택(Surface/테두리 Tertiary) · 선택(Primary 연틴트/테두리 Primary/체크)

---

### [S-04] 온보딩 Step 3 — 전화 시간 선택

```
[Progress 3/4] [← 뒤로]
  [Heading 1] 언제 전화 드릴까요?
  [Body/Secondary] 매일 이 시간에 AI가 알림을 보내요.
  [드럼롤 Picker] 기본 22:00 · 분 단위 00/30
  [추천 Chip — 가로 스크롤]
    "퇴근 후 🌆 19:00" / "저녁 후 🍽️ 20:00" / "잠들기 전 🌙 22:00"(기본) / "자정 전 ✨ 23:30"
  하단 고정 [Primary] 다음
```

---

### [S-05] 온보딩 Step 4 — 0일차 미니 통화

**목적:** 첫 데이터 확보 + "이런 느낌이구나" 즉시 체험 → 이탈 방지 + 데모 즉시 시연.

```
[Progress 4/4]
  [Heading 1] 처음 만났으니 가볍게 인사해요 👋
  [Body/Secondary] 딱 하나만 물어볼게요. 누르면 {name}님과의 첫 대화가 시작돼요.
  [일러스트 — 전화기/빛 퍼짐]
  하단 고정 [Primary] 첫 대화 시작하기
```

## 9. 데이터 모델

### users

```sql
id              UUID PK
name            TEXT NOT NULL          -- 표시 이름
nickname        TEXT NOT NULL          -- AI가 부를 호칭
call_time       TIME NOT NULL          -- 매일 전화 시간(로컬)
timezone        TEXT DEFAULT 'Asia/Seoul'
voice_tone      ENUM('calm','bright','plain') DEFAULT 'calm'
push_token      TEXT                   -- FCM 토큰 (전화번호 대체)
created_at      TIMESTAMPTZ
```

> 🔧 `phone` 컬럼 제거 — 인앱 푸시는 `push_token`으로 충분. (계정 식별 방식은 §14-B)

### conversations

```sql
id                UUID PK
user_id           UUID FK → users.id
called_at         TIMESTAMPTZ
duration_sec      INTEGER
call_length       ENUM('1min','3min','7min')
status            ENUM('completed','missed','in_progress','dropped')
input_mode        ENUM('voice','text','mixed')
transcript        JSONB                -- [{role, timestamp, text}]
summary           TEXT                 -- 한 줄 요약
keywords          TEXT[]               -- 최대 3
emotion           TEXT
emotion_intensity INTEGER              -- 1~5
key_scene         TEXT
hidden_desire     TEXT
ai_comment        TEXT
rating            INTEGER              -- 1~5 (NULL=미평가)
rating_tags       TEXT[]               -- 원탭 칩
rating_comment    TEXT
created_at        TIMESTAMPTZ
```

### action_items (실천 루프)

```sql
id                UUID PK
conversation_id   UUID FK → conversations.id
user_id           UUID FK → users.id
content           TEXT NOT NULL
scheduled_for     DATE                 -- 보통 다음 날
status            ENUM('pending','done','rescaled','skipped')
rescaled_content  TEXT
reviewed_at       TIMESTAMPTZ          -- 다음 통화 확인 시각
created_at        TIMESTAMPTZ
```

### call_schedules

```sql
id            UUID PK
user_id       UUID FK → users.id
scheduled_at  TIMESTAMPTZ
retry_count   INTEGER DEFAULT 0
status        ENUM('pending','sent','snoozed','expired')
snoozed_until TIMESTAMPTZ
created_at    TIMESTAMPTZ
```

> streak·평균 별점·실천 달성률 = `conversations`/`action_items`에서 **파생 계산**(별도 테이블 없음).
> 음성 원본(raw audio) 저장 여부는 §13 프라이버시 결정 사항.

### 10-2. 대화 기록

**GET /conversations** — Query: `limit, cursor, filter(week|month|all)`

```json
{ "conversations":[{ "id":"uuid","called_at":"...","duration_sec":452,
  "call_length":"3min","status":"completed","summary":"오늘은 업무 스트레스와…",
  "keywords":["번아웃","주말계획","운동"],"emotion":"지침","emotion_intensity":3,

  "next_cursor":"uuid" }
```

**GET /conversations/{id}** — 위 필드 + `transcript`, `key_scene`, `hidden_desire`, `ai_comment`, `rating_tags`, `rating_comment`, 전체 `action_item`

**PATCH /conversations/{id}/rating** — `{ "rating":4, "tags":["질문이 좋았어요"], "comment":"…" }`

### 10-3. 실천 루프

**PATCH /action-items/{id}** — `{ "status":"done" }` (done|skipped)

**GET /action-items/pending** — 진행 중 과제 1건

### 10-4. 전화 스케줄

**GET /schedule/next** — `{ "next_call_at":"…","minutes_remaining":142 }`

**POST /schedule/snooze** — `{ "snoozed_until":"…" }` (1시간 뒤)

### 10-5. 음성 대화 (WebSocket)

> **음성 스택과 무관하게 동일 인터페이스.** 서버가 내부에서 파이프라인 또는 Gemini Live로 라우팅(§11-6).

**POST /conversations/start** (WS 연결 전)

```json
// Request
{ "call_length":"3min" }
// Response 201
{ "session_id":"uuid","conversation_id":"uuid",
  "pending_action_item":{ "content":"동료한테 말 걸어보기","scheduled_for":"2026-06-07" } }
```

**WS /ws/conversation/{session_id}**

```
클라이언트 → 서버:
  {type:"audio_chunk", data:<base64 PCM>}
  {type:"end_of_speech"}
  {type:"end_call"}
  {type:"text_input", text}

서버 → 클라이언트:
  {type:"transcript", role, text}
  {type:"ai_response_start"}
  {type:"audio_chunk", data}
  {type:"ai_response_end", text}
  {type:"conversation_ended", session_id}
```

## 11. AI 대화 시스템 스펙

### 11-1. 4블록 대화 프레임

| 블록 | 목적 | 예시 |
| --- | --- | --- |
| ① 감정 체크 | 오늘 감정 상태 | "오늘 기분을 한 단어로 표현하면요?" |
| ② 오늘의 장면 | 구체적 사건 | "오늘 가장 오래 마음에 남은 순간은요?" |
| ③ 의미/해석 | 감정 뒤 욕구 | "그 감정 뒤에 어떤 바람이나 실망이 있었을까요?" |
| ④ 내일의 실천 | 행동 실험 | "아주 작게, 내일 해볼 수 있는 것 하나만 정해볼까요?" |

길이별: 1분=①④ / 3분=①②③④ / 7분=①②+깊이③+렌즈④

### 11-2. 변주 엔진 (식상함 방지)

②③ 블록에 **회전 렌즈** 매일 다르게 적용:

1. 감사 / 2. 관계 / 3. 도전 / 4. 몸 / 5. 성취 / 6. 욕구 / 7. 쉼

(요일 기반 순환, 과거 기록과 겹치면 건너뜀)

**과거 참조(3회+ 누적 시 활성):** "지난주에 운동하면 풀린다 하셨잖아요, 오늘은 좀 움직이셨어요?"

### 11-3. 실천 루프 프롬프팅

**전날 과제 있을 때 첫마디:** "안녕하세요 {nickname}님! 어제 {content} 해보기로 했었잖아요, 어떻게 됐어요?"

- 했음 → "잘하셨어요! 어떤 느낌이었어요?" → 새 실천
- 못 함 → "괜찮아요, 뭐가 걸렸어요?" → 더 작게 쪼개기(rescaled)
- 부분 → "그것도 충분해요!" → 이어가기

### 11-4. 시스템 프롬프트 (요지)

```
당신은 여보세요의 AI 대화 파트너입니다.
컨텍스트: 호칭 {nickname} / 날짜 {date} / 길이 {call_length} / 렌즈 {today_lens}
        / 이전 실천 {pending_action_item} / 최근 요약 {last_3_summaries}
역할: 따뜻한 경청자이자 자기 탐구 파트너. 심리상담사가 아니라 옆에서 같이 생각해주는 존재.
원칙:
1) 한 번에 질문 하나만.
2) 공감 먼저, 질문은 다음.
3) 절대 단정 금지. ("그건 번아웃이에요"❌ → "그 피로감이 언제부터였을까요?"✅)
4) 의학적 진단·치료 권고 금지.
5) 마무리 신호 시 자연스럽게 클로징.
실천(④): 그날 대화 속 구체 상황 기반, 아주 작고 구체적으로. 사용자가 말하게 유도 후 정리·확인.
안전: 위기 신호(자해·자살 등) 감지 시 §13 안전 프로토콜로 전환.
클로징: "오늘도 이야기 나눠줘서 고마워요. {핵심 한 줄}. 내일 {실천} 어떻게 됐는지 들을게요. 잘 쉬세요 {nickname}님."
```

### 11-5. 요약 생성 프롬프트 (통화 종료 후, 비실시간)

```
입력: {nickname}님과의 대화 transcript
출력(JSON):
{
  summary,          // 40자 이내
  keywords,         // 최대 3개
  emotion,          // 한 단어
  emotion_intensity,// 1~5
  key_scene,
  hidden_desire,
  ai_comment,
  action_item       // 없으면 null
}
```

> 요약은 실시간이 아니므로 **상위 모델**(품질 우선) 사용 가능.

### 11-6. 실시간 음성 아키텍처 — **결정 항목** ⭐

음성 레이어를 **추상화(VoiceEngine 인터페이스)**해서, 아래 두 구현 중 선택해도 §10 WS 인터페이스·앱은 그대로.

**Option A — STT → LLM → TTS 파이프라인** *(현 기본안)*

```
[마이크] → WS → [STT 스트리밍] → [대화 관리(FastAPI): 턴/컨텍스트/블록]
        → [LLM 스트리밍(저지연 모델)] → [TTS 스트리밍] → WS → [스피커]
```

- STT: Gemini 3 Flash / GPT-4o Transcribe
- LLM: Gemini 3.1 Flash Lite → fallback GPT-4o mini
- TTS: Google Chirp 3 / GPT-4o-mini-tts
- ✅ 컴포넌트 교체·최적화 자유, 텍스트 단계 제어(가드레일·요약·로깅), 톤 커스텀, 한국어 개별 튜닝
- ⚠️ 지연 누적(직렬), barge-in/턴테이킹 구현 부담, 오케스트레이션 복잡

**Option B — Gemini Live (실시간 native audio)** *(검토 중)*

```
[마이크] → WS → [서버 프록시] → [Gemini Live audio↔audio] → WS → [스피커]
                                  └ transcript 별도 수신 → 요약/로깅
```

- ✅ 저지연·자연스러운 흐름·끼어들기, 오케스트레이션 단순, 빌드 빠름
- ⚠️ 텍스트 중간 제어 약함(가드레일·요약 별도 설계), 음성/톤 커스텀 제약, 모델 종속

**목표 레이턴시:** 사용자 말 끝 → AI 음성 시작 **≤ 2.5초**

**권장:** 착수 전 **두 옵션 한국어 품질·지연·비용 스파이크 테스트** → 결정(§14-B2). VoiceEngine 추상화로 전환 비용 최소화.

## 13. 에러 · 엣지 케이스 · 안전

### 13-1. 대화 관련

| 케이스 | 처리 |
| --- | --- |
| 마이크 권한 거부 | 안내 모달 + 설정 이동 + **텍스트 모드로 통화** |
| WebSocket 끊김 | 자동 재연결 3회 → 실패 시 토스트 + 재연결 버튼 |
| STT 인식 실패 | AI: "잘 안 들렸어요, 다시 말씀해주실 수 있나요?" |
| 통화 30초 미만 종료 | summary 생략, status=missed |
| 통화 중 앱 종료/네트워크 끊김 | status=dropped, 지금까지 내용으로 부분 요약 |
| 모델 타임아웃 | 준비된 fallback 멘트("잠시만요…") |
| TTS 지연 | 파형 느리게 유지, 준비되면 재생 |
| 모델 부적절 응답 | 시스템 프롬프트 가드레일 + 종료 후 신고 채널 |

### 13-2. 앱 관련

| 케이스 | 처리 |
| --- | --- |
| 앱 백그라운드 알림 | FCM 푸시 → 탭 시 앱 열리며 오버레이 |
| 푸시 권한 거부 | 앱 내 알림 배너로 대체 |
| 네트워크 없음 | 캐시된 최근 카드 표시 + 오프라인 배너 |
| 카드 생성 실패 | "다시 불러오기" 재시도 |
| 세션 만료 | Supabase 자동 갱신 → 실패 시 재로그인 |

### 13-3. 시간대

- 서버 UTC 저장 · 표시 Asia/Seoul(MVP 한국) · 스케줄러: `call_time + tz → UTC → APScheduler`

### 13-4. 🛟 안전 (위기 대응) — **MVP 필수**

- **위기 신호(자해·자살·심각한 정서 위기) 감지 → 대화 톤 전환:** 공감 + 증폭/반추 회피 + **전문 자원 안내**(예: 자살예방상담전화 109 — *시행 전 최신 번호·자원 검증 필수*) + "상담 대체가 아님" 고지
- 안전 모드에선 평가 질문·종료 압박 회피
- 책임 한계·고지 문구 위치 = 온보딩 + 설정(§14-E)
- **프라이버시:** 음성/transcript = 민감정보 → 저장 동의·암호화·보관기간·삭제(설정 내 "데이터 삭제") 필수. 음성 원본 저장 여부 §14-E.

1. **인앱 콜 확정**(제안: 확정). 추후 VoIP/CallKit 네이티브 콜 화면 고려할지.

5. **계정 식별/Auth 방식** — 전화번호 제거 후 소셜/이메일/익명 중?

3. 측정 인프라 — 응답률·발화량·리텐션 로깅(분석 툴).
